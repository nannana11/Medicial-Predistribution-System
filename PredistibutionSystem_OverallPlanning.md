# 人工智能辅助医院门诊预分诊系统开发思路

## 一、项目概述

本项目计划开发一个“人工智能辅助医院门诊预分诊系统”，用于部署在医院门诊大厅、自助服务机、咨询台等场景中，帮助患者根据自身症状描述获得初步的就诊科室建议。

系统整体采用 **客户端-服务器架构**：

* 客户端部署在医院自助服务机或咨询台电脑上，负责与用户交互。
* 服务器部署在医院内网服务器上，负责接收请求、调用大模型 API、处理分诊逻辑并返回结果。
* 大模型作为系统中的辅助推理模块，用于根据用户输入生成预分诊建议。
* 后续可加入数据库、规则引擎、管理后台、日志审计、安全校验等模块，使系统逐步接近真实医院使用场景。

本系统的定位不是“AI 诊断系统”，而是“门诊预分诊辅助系统”。系统只负责提供科室推荐、风险提醒和必要追问，不替代医生诊断，不提供明确疾病判断和具体用药建议。

---

## 二、项目目标

### 2.1 总体目标

开发一个能够在医院内网环境下运行的智能预分诊系统，实现以下核心能力：

1. 用户可以在客户端输入症状描述。
2. 客户端将用户输入发送给服务器。
3. 服务器调用大模型 API，生成预分诊建议。
4. 服务器将结果返回给客户端。
5. 客户端以清晰、友好的方式展示推荐科室、紧急程度和提示信息。
6. 后续逐步加入多轮对话、数据库记录、安全规则、后台管理和部署优化。

### 2.2 第一阶段目标

第一阶段主要完成系统的基础闭环：

> 用户在客户端输入文字 → 客户端发送给服务器 → 服务器调用大模型 API → 服务器返回结果 → 客户端展示结果。

第一阶段不追求复杂医疗逻辑，不追求完整数据库设计，不追求精美 UI，而是先保证整个系统从客户端到服务器再到大模型 API 的流程能够跑通。

## 三、系统总体架构

### 3.1 架构示意

```text
用户
 ↓
JavaFX 客户端 / 自助服务机客户端
 ↓ HTTP / JSON
Spring Boot 服务器
 ↓
Prompt 构造模块
 ↓
大模型 API 调用模块
 ↓
DeepSeek / 其他大模型 API
 ↓
Spring Boot 服务器处理返回结果
 ↓
JavaFX 客户端展示结果
```

整个系统主要分为以下几个部分：

```text

├── triage-client/       # 客户端，JavaFX 项目
├── docs/                # 项目文档
├── scripts/             # 部署或启动脚本
└── README.md            # 项目说明
```

### 5.1.1 Controller 层

`TriageController` 负责对外提供接口。

主要接口：

```text
POST /api/triage/message
```

职责：

1. 接收客户端传来的用户症状描述。
2. 调用 `TriageService`。
3. 将结果返回给客户端。

---

### 5.1.2 Service 层

`TriageService` 是核心业务模块。

职责：

1. 接收用户输入。
2. 调用急诊规则检查模块。
3. 构造 Prompt。
4. 调用大模型服务。
5. 整理返回结果。
6. 返回给 Controller。

第一阶段流程：

```text
用户输入
 ↓
TriageController
 ↓
TriageService
 ↓
PromptBuilder
 ↓
LlmService
 ↓
DeepSeekClient
 ↓
返回结果
```

---

### 5.1.3 PromptBuilder 模块

`PromptBuilder` 用于构造发送给大模型的 Prompt。

第一阶段可以先使用固定 Prompt：

```text
你是医院门诊预分诊辅助系统，不是医生。
你的任务是根据用户描述推荐合适的就诊科室，并判断是否可能需要急诊。
你不能进行明确疾病诊断，不能提供具体用药建议，不能替代医生判断。
请用简洁中文回答。
用户输入：{{USER_INPUT}}
```

后续阶段可以升级为结构化 Prompt，要求模型返回 JSON。

---

### 5.1.4 LlmService 模块

`LlmService` 用于封装大模型调用逻辑。

职责：

1. 接收 Prompt。
2. 调用 `DeepSeekClient`。
3. 处理模型返回内容。
4. 发生异常时返回友好错误信息。

设计原则：

```text
业务代码不要直接调用 DeepSeek API。
业务代码只调用 LlmService。
LlmService 再调用具体的大模型客户端。
```

这样后续如果从 DeepSeek 换成其他模型，不需要大改业务代码。

---

### 5.1.5 DeepSeekClient 模块

`DeepSeekClient` 专门负责与 DeepSeek API 通信。

职责：

1. 构造 HTTP 请求。
2. 添加 API Key。
3. 发送请求。
4. 接收响应。
5. 处理 API 调用失败、超时、限流等问题。

API Key 不应写死在代码中，应该放在配置文件或环境变量中。

示例：

```yaml
llm:
  deepseek:
    api-key: ${DEEPSEEK_API_KEY}
    base-url: https://api.deepseek.com
    model: deepseek-chat
```

---

### 5.1.6 EmergencyRuleChecker 模块

第一阶段可以先简单保留该模块，后续逐步增强。

职责：

1. 检查用户输入中是否存在高危症状。
2. 如果命中高危规则，可以直接返回急诊提示。
3. 避免完全依赖大模型判断急危重症。

示例高危关键词：

```text
胸痛
呼吸困难
意识不清
大出血
抽搐
自杀
剧烈头痛
孕妇腹痛
儿童高热惊厥
```

第一阶段可以不做复杂规则，但建议提前保留该模块，方便后续扩展。

## 七、第二阶段：结构化分诊结果

第一阶段完成后，第二阶段重点是让系统输出更加稳定、可解析、可展示的结构化结果。

### 7.1 第二阶段目标

将模型自由文本回复升级为固定 JSON 格式。

推荐返回格式：

```json
{
  "recommendedDepartment": "呼吸内科",
  "alternativeDepartments": ["急诊科"],
  "urgencyLevel": "medium",
  "needEmergency": false,
  "needFollowUp": true,
  "followUpQuestions": [
    "症状持续了几天？",
    "是否有明显呼吸困难？",
    "体温最高是多少？"
  ],
  "reason": "用户描述咳嗽、发热、胸闷，优先考虑呼吸系统相关问题。",
  "patientMessage": "根据您的描述，建议优先咨询呼吸内科。如胸闷明显或呼吸困难，请及时前往急诊。",
  "safetyNotice": "本结果仅用于门诊预分诊参考，不能替代医生诊断。"
}
```

### 7.2 第二阶段任务

1. 修改 Prompt，要求模型严格返回 JSON。
2. 后端解析 JSON。
3. 后端校验字段是否合法。
4. 客户端将不同字段展示在不同区域。
5. 支持推荐科室、紧急程度、追问问题分别展示。

---

## 八、第三阶段：多轮对话与会话管理

### 8.1 阶段目标

让系统支持用户连续对话，而不是每次都当成新问题。

例如：

```text
用户：我头疼
系统：请问头疼持续多久？是否发热？
用户：两天了，有点发烧
系统：建议咨询神经内科或发热门诊……
```

### 8.2 核心任务

1. 为每次咨询创建 `sessionId`。
2. 保存当前会话中的历史消息。
3. 每次请求模型时携带必要历史上下文。
4. 客户端支持聊天式展示。
5. 后端限制最大历史长度，避免 Prompt 过长。

### 8.3 会话数据结构

```json
{
  "sessionId": "abc123",
  "messages": [
    {
      "role": "user",
      "content": "我头疼"
    },
    {
      "role": "assistant",
      "content": "请问头疼持续多久？是否发热？"
    }
  ]
}
```

## 九、第四阶段：安全规则与风险控制

医疗场景不能完全依赖大模型，因此需要加入规则层和校验层。

### 9.1 阶段目标

提高系统安全性，避免大模型输出危险、不合适或不稳定内容。

### 9.2 核心模块

```text
EmergencyRuleChecker
TriageResultValidator
SensitiveInfoFilter
DepartmentValidator
```

### 9.3 安全规则

需要重点识别以下情况：

```text
胸痛伴大汗
呼吸困难
意识不清
大出血
严重外伤
抽搐
突发肢体无力
孕妇剧烈腹痛
儿童高热惊厥
自杀或自伤倾向
```

如果命中高危规则，可以直接返回：

```text
建议立即联系现场医护人员或前往急诊科。
```

### 9.4 输出校验

后端需要检查模型输出：

1. 是否是合法 JSON。
2. 推荐科室是否在医院科室列表中。
3. 紧急程度是否合法。
4. 是否出现明确诊断。
5. 是否出现具体用药建议。
6. 是否出现“不用去医院”等不当表述。

### 10.3 注意事项

真实医院场景中，系统不应随意保存患者姓名、身份证号、手机号等敏感信息。第一版建议只保存匿名会话记录和症状文本，后续如果涉及真实医疗数据，需要进一步考虑隐私保护、权限控制和合规要求。

## 十二、第七阶段：部署与运行

### 12.1 第一阶段部署方式

第一阶段可以使用局域网部署：


## 十五、第一阶段推荐完成路径

第一阶段建议严格按照以下顺序完成：

```text
1. 创建 Spring Boot 后端项目
2. 写一个假的 /api/triage/message 接口
3. 创建 JavaFX 客户端项目
4. 客户端调用后端假接口
5. 后端接入 DeepSeek API
6. 后端返回模型结果
7. 客户端展示真实结果
8. 加入输入为空、连接失败、API 失败等错误处理
9. 完成局域网测试
10. 整理 README 和项目截图
```

第一阶段最重要的不是功能复杂，而是完成完整闭环。

只要第一阶段跑通，后续所有功能都是在这个闭环上逐步增强。

## 十七、当前最优先任务

当前阶段最优先完成的是：

```text
Spring Boot 服务器 + JavaFX 客户端 + 大模型 API 调用
```

第一阶段的核心闭环为：

```text
用户输入
 ↓
JavaFX 客户端
 ↓
Spring Boot 服务器
 ↓
PromptBuilder 构造 Prompt
 ↓
DeepSeekClient 调用 API
 ↓
TriageService 整理结果
 ↓
JavaFX 客户端展示
```

这个闭环一旦完成，项目就从想法进入了真正可运行的工程原型阶段。
